AI Is Not a Magician ü™Ñ

Bear with me through our baseline here‚Äîeverything that follows depends on understanding this distinction first.

I jump into so many random debates‚Äîusually uninvited‚Äîabout the ultimate usefulness of AI and nearly every single argument I hear about AI behaving badly gets the exact same reply from me:

Of course it torched your entire repo‚Äîand probably in record time! You just let an unsupervised mostly unhinged guessing machine loose with a flamethrower, blanket auto-approvals, and admin-level control. üî•üêâü§Ø

To understand why that keeps happening, you have to understand the difference between what we casually call ‚ÄúAI‚Äù and the large language model (LLM) underneath it. These are not the same thing and shouldn‚Äôt be treated like they are.

Every LLM is stateless by design, meaning every call you make is a clean slate. You give it input and it outputs a response. Conceptually, it's no different than an HTTP call‚Äîexcept instead of returning a standardized value with a predictable schema, the LLM returns something more... creative.

The AI system sitting on top of that model is what makes everything feel connected. That's the system deciding what context to attach, what history to include, and how to frame every request so the LLM has any chance of responding in a useful way. If the AI fails at managing that data, the LLM never stood a chance to begin with.

Last I heard, GitHub supports something like 180 million users. If I guess and say 80% of them use Copilot, that‚Äôs 144 million different user workflows‚Äîand therefore 144 million competing definitions of what a ‚Äúgood‚Äù response looks like.

It is not designed to magically work out of the box‚Äîespecially in production codebases‚Äîno matter how much they'd like you to believe otherwise!

Instructions Are a Priority Stack üß±

One of the very first posts I wrote‚Äîwhich honestly deserves an update‚Äîwas about setting up custom repo instructions. There‚Äôs no shortage of instruction-writing advice floating around out there, and there's just as many opinions about the ‚Äúright‚Äù way to do it.

I'm not even remotely invested in that particular debate‚ÄîI know, I was shocked, too!‚ÄîI do know, beyond the shadow of a doubt, that if you expect AI to play by your rules, then you first have to explicitly tell it what those rules are.

I will say it again: AI is not a coding magician. It‚Äôs also not a particularly great guesser. The system instructions in VS Code do a decent job of orienting the model toward the idea that "you write code", but they're also intentionally generic so it works for everyone.

Here‚Äôs a snippet so you can see what I mean:

The full set of system instructions includes:

  • JSON for every enabled tool
  • well over a hundred lines of system instructions
  • all global user instructions
  • all applicable repo instruction paths
  • custom agent names and metadata

The ordering matters. LLMs will start summarizing aggressively after processing the first chunk of input. If something matters, put it where it‚Äôs least likely to be compressed. The data volume matters too‚Äîmore text usually means more summarization, not more intelligence.

And if you introduce instructions that directly conflict with the system-level ones, the results don‚Äôt get better. They get progressively worse.

ü¶Ñ Telling AI it's an "expert coder" is largely unnecessary. One: because that's already been done for you by the system. Two: experience‚Äîand a healthy side of gut instinct‚Äîtells me those "expert" statements are doing more harm than good. Personally, I stopped using any variant of the "expert" statement a long time ago.

People ask me why ‚Äúmy AI‚Äù works consistently, and the answer is always some version of: because I learned how to write instructions and adapt them to a version the system can reliably manage.

This is my personal baseline. These instructions reflect my hardware, workflows, tool choices, and even the personality adjustment is designed specifically to avoid my instant-rage trigger during long pairing sessions. Copilot is my primary use case, but these rules are wired everywhere I work and I use them in real projects.

This is definitely not the GitHub-marketing version of AI that exists to flatter the user. I gave it strict boundaries and strong opinions on purpose. I want pushback. In practice, Copilot waters that down way more than I‚Äôd like. So, perfecting this personality is going to be a work in progress for the foreseeable future.

Trust Is Earned, Validation Is Mandatory üß™

Whenever AI touches code‚Äîwhether it‚Äôs a new feature or a quick fix‚Äîthe results have to pass a specific set of checks before anything is presented to me in chat.

Not every repo is identical, so I've started using a Makefile in all of my personal projects. That gives AI a single, explicit definition of what ‚Äúvalidation‚Äù means. Without that, it will go looking for standards on its own‚Äîand when it inevitably can‚Äôt find them, it guesses. My instructions make that behavior explicit so it defaults to the simplest path instead of inventing a new maze of random bash scripts just to run a missing lint command. üôÑ

Note that there‚Äôs nothing remotely deterministic about asking any AI agent to run its own validations. Do not expect perfection every turn‚Äîyou will be disappointed! The real solution requires more system-level support than is currently available, though.

This setup is my temporary placeholder for the smart agent system that ultimately just works. We‚Äôll get there‚Äîeventually. In the meantime, this helps:

<div class="highlight js-code-highlight"> 
<span class="gu">### Mandatory Verification Loop (Bounded, With Escape Hatch)</span>
<span class="p">
-</span> Before responding with code or implementation changes, run a <span class="gs">**validation loop**</span> covering:
<span class="p">-</span> formatting and linting
<span class="p">-</span> tests executed and passing
<span class="p">-</span> coverage reviewed
<span class="p">-</span> documentation updated (when relevant)
<span class="p">-</span> security implications considered
<span class="p">-</span> solution simplicity verified

<span class="gs">**Tool Preference**</span>: When <span class="sb">`make ai-checks`</span> exists in the repo, prefer it over ad-hoc validation commands.
<span class="p">
-</span> <span class="gs">**Maximum iterations: 3 total attempts.**</span>

Kill the Default Personality üî™

The very first thing I do with any new system is kill off the default ‚Äúhelpful‚Äù personality. If I wanted a behavioral therapist to tell me how great I am, I wouldn‚Äôt be writing software every day. It only takes one ‚ÄúYou‚Äôre absolutely right!‚Äù response for me to be done with the niceties‚Äîand that‚Äôs still one too many!

I also don‚Äôt want a play-by-play of which files were updated or a long explanation of how the solution was reached. If I didn‚Äôt explicitly ask, then I genuinely do not care. The moment a small essay starts forming in chat, I‚Äôm out‚Äîthe IDE gets minimized and I just wait for it to finish embarrassing itself. I‚Äôm not reading past the last few lines, and I‚Äôve absolutely burned more than one prompt asking a different model to summarize the response into something I might actually comprehend.

My go-to AI personality is not designed for the easily offended or for anyone in a ‚Äútrying to learn something new‚Äù headspace. Instead, it does this:

Leave Git Alone‚ÄîIt Belongs to Me üîê

I do occasionally throw AI the keys and sit back just to see which fireworks fly and where the system cracks. Those repos are set up as explicit experiments and designed for that purpose from the start‚Äîit‚Äôs never the baseline.

In my normal workflows, AI is leashed far away from anything that writes to either Git or GitHub. Inside the IDE, source control staging is my truth for code I‚Äôve already reviewed. The moment Copilot adds to it, I'm no longer certain of what was reviewed versus not, which means starting over.

I do everything I can to keep Git history pristine, which means AI doesn‚Äôt touch it beyond read-only commands or research in external repos. The --no-pager rule is a bonus I added after AI kept getting stuck waiting for input any time it tried to view a diff.

<span class="p">-</span> Never commit.
<span class="p">-</span> Never push.

Config Has Boundaries, Too üöß

AI does not touch my repo configuration without explicit authorization. Ever. This is a direct extension of my ongoing mission to eliminate every // eslint-disable-next-line that‚Äôs ever been tossed into a repo just to force a green check. More importantly, it prevents AI from quietly reproducing the exact patterns I‚Äôm trying to get rid of in the first place.

If a config change would genuinely help‚Äîand isn‚Äôt being used to paper over a failure‚ÄîAI is expected to surface the suggestion clearly in chat. That way, my formatters and linters don‚Äôt become useless because all the rules were disabled while I wasn‚Äôt paying attention.

<div class="highlight js-code-highlight"> 
<span class="gu">### Repository Configuration Boundaries</span>
<span class="p">
-</span> You may <span class="gs">**not**</span> modify repository configuration files unless explicitly instructed.
<span class="p">  -</span> This includes: dotfiles, package.json, pyproject.toml, tsconfig.json, eslint configs, prettier configs, etc.
<span class="p">  -</span> This applies to files that <span class="gs">**control or maintain the repo itself**</span>.
<span class="p">  -</span> This does <span class="gs">**not**</span> include code or documentation the repo is designed to provide.
<span class="p">-</span> You <span class="gs">**must**</span> surface recommended config changes clearly in chat when they would improve correctness, safety, or consistency.
<span class="p">  -</span> Suggestions are expected.
<span class="p">  -</span> Silent edits are forbidden.

Principles Over Convenience ü™®

Some of these instructions are intentionally written to counteract specific, ultra-annoying AI tendencies‚Äîlike curbing Claude‚Äôs occasional bout of what I can only describe as ‚Äúexcessive compulsive disorder.‚Äù

Most of what I build is either a toy or a dev utility. If something changes, then it changes. I have zero interest in complicating otherwise clean systems with backwards compatibility‚Äîespecially when the only user is me.

I‚Äôm also deeply addicted to automation, even when the only real payoff is perfectly numbered releases starting from zero. Breaking changes are recorded accurately in commits using a reusable AI prompt (also in my awesome-github-copilot repo). Release-please watches main, handles the semver bump on merge, and generates a changelog tied to an immutable GitHub release.

Boring. Predictable. Functional. Perfect.

<div class="highlight js-code-highlight"> 
<span class="gu">### Non-Negotiable Principles of Development</span>
<span class="p">
-</span> <span class="gs">**KISS**</span> and <span class="gs">**YAGNI**</span> outrank all other design preferences.
<span class="p">-</span> The diff should be:
<span class="p">  -</span> minimal
<span class="p">  -</span> intentional
<span class="p">  -</span> easy to reason about
<span class="p">-</span> <span class="gs">**Backward compatibility is forbidden unless explicitly requested.**</span>
<span class="p">  -</span> Do not preserve old behavior ‚Äújust in case.‚Äù
<span class="p">  -</span> Do not carry dead paths.
<span class="p">  -</span> If it no longer exists, it only belongs in the commit message explanation.
<span class="p">-</span> <span class="gs">**Prerelease changes never constitute a breaking change.**</span>

Docs Are a Tool, Not a Diary ‚úçÔ∏è

Documentation exists to be useful. The problem is that nobody ever defined what ‚Äúuseful‚Äù means for the AI that‚Äôs now writing it. And what does AI do when it doesn‚Äôt have a clear answer? It guesses‚Äîand it usually guesses that you wanted everything documented from every possible angle across the entire codebase.

Spoiler: that‚Äôs never actually helpful.

On top of that, I‚Äôm convinced most of us are conditioned to ignore even the best-written docs by default. Don‚Äôt believe me? When was the last time you were asked an extremely technical question and your first thought was, ‚ÄúI bet that‚Äôs accurate, up-to-date, and easy to find in the documentation‚Äù? ü§∑‚Äç‚ôÄÔ∏è

Which leaves exactly zero reasons to let AI free-style pages of prose for fun. Instead, you have to tell it what documentation is for:

<div class="highlight js-code-highlight"> 
<span class="gu">### Documentation Rules</span>
<span class="p">
-</span> Use <span class="gs">**Mermaid**</span> for all diagrams:
<span class="p">  -</span> Include accessibility labels
<span class="p">  -</span> Initialize using the <span class="gs">**default profile**</span>
<span class="p">  -</span> Always validate diagram syntax with available tools
<span class="p">  -</span> Prefer deterministic, non-interactive rendering
<span class="p">-</span> Update <span class="gs">**existing documentation**</span> whenever possible.
<span class="p">-</span> ADRs are historical artifacts and must not be rewritten.
